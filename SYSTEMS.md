## 1. Overview

```
                         +--------------------+
                         |   Data Scraper     |
                         |  (data_scraper/)   |
                         |  FastAPI + SQLite  |
                         +--------+-----------+
                                  |
                          POST /api/v1/places
                           (sync scraped data)
                                  |
                                  v
+------------------+     +--------------------+     +------------------+
|   Web App        |     |   Main Server      |     |   Mobile App     |
|  (apps/web/)     +---->|    (server/)       |<----+  (apps/mobile/)  |
|  React + Vite    |     |  FastAPI + SQLite  |     |  Expo + RN       |
|  Tailwind CSS    |     |  SQLModel + JWT    |     |  React Navigation|
+------------------+     +--------------------+     +------------------+
        |                         |                         |
        |        REST API (JSON, Bearer JWT)                |
        +-------------------------+--------------------------+
```

**Three backend services:**
- **Main Server** (`server/`) -- Core API serving all client apps
- **Data Scraper** (`data_scraper/`) -- Ingests place data from Google Maps and Google Sheets, syncs to main server

**Two frontend clients:**
- **Web App** (`apps/web/`) -- Vite + React SPA with Tailwind CSS
- **Mobile App** (`apps/mobile/`) -- Expo + React Native

All entities use stable, autogenerated `*_code` identifiers (never numeric IDs) across the entire stack.

#### Dynamic Attribute System (EAV)

| # | Model                         | Primary Key / Identifiers                | Description                                                    |
|---|-------------------------------|------------------------------------------|----------------------------------------------------------------|
| 12| **PlaceAttributeDefinition**  | `attribute_code`                         | name, data_type, icon, label_key, is_filterable, is_specification, religion, display_order |
| 13| **PlaceAttribute**            | Unique: place_code + attribute_code      | value_text, value_json (JSON) -- stores actual attribute values |

The EAV (Entity-Attribute-Value) pattern allows flexible, religion-specific metadata on places without schema changes. Attribute definitions describe what attributes exist (e.g., "prayer_times" for Islam, "service_times" for Christianity, "deities" for Hinduism). PlaceAttribute stores the actual values per place.

### Key Server Features

- **Dynamic Attribute System (EAV)**: PlaceAttributeDefinition + PlaceAttribute tables enable flexible, religion-specific metadata without schema migrations. Bulk attribute fetching performs a single query for multiple places.
- **Religion-Specific Timings**: Islam (prayer times), Christianity (service times), Hinduism (deities) -- all driven by attribute definitions.
- **Haversine Distance Calculation**: Places can be sorted by proximity to user coordinates.
- **Opening Hours Detection**: Supports per-day format and legacy format parsing.
- **External Reviews**: Google Maps reviews imported with `source="external"` and `author_name`.
- **Image Blob Storage**: Images stored as base64-encoded blobs in the database, served via a dedicated endpoint.
- **JWT Authentication**: Bearer token auth with bcrypt password hashing.
- **Password Reset Flow**: Token-based reset (email sending is a stub in development).
- **Check-In System**: Users check in to places with optional notes and photos. Supports "this month" and "on this day" historical queries.
- **Favorites System**: Simple toggle per user per place.
- **Groups**: Private/public groups with invite codes, ordered path of places (`path_place_codes`), member roles, leaderboard (ranked by check-ins), and activity feed.
- **Notifications**: Types include `check_in`, `group_invite`, `group_join`, `review`. JSON payload for flexible content.
- **Internationalization**: English (default), Arabic (RTL), Hindi. All translations served from the backend. English fallback when a key is missing.
- **Seed Data**: 10 real pilgrimage sites worldwide, 15 attribute definitions, full translation dictionaries, sample users, groups, and reviews.

### Scrapers

**Google Maps Scraper:**
- Grid-based geographic search within defined boundaries
- Auto-detects religion from Google place types (mosque -> Islam, church -> Christianity, temple -> Hinduism)
- Downloads place images as base64 blobs
- Extracts reviews and editorial descriptions
- Configurable via PlaceTypeMapping for which Google types to search

**Google Sheets Scraper:**
- Imports place data from configured Google Sheets
- Sheet ID and range stored in DataLocation config

### Sync Mechanism

Scraped places are synced to the main server one-by-one via `POST /api/v1/places`. The main server's create-or-update logic handles deduplication. This keeps the scraper decoupled from the main server's database.

## 6. Shared Features

These features are implemented consistently across the full stack.

### Internationalization (i18n)

- **Languages**: English (default), Arabic, Hindi
- **RTL Support**: Enabled automatically when locale is Arabic (`ar`)
- **Architecture**: All UI strings served from the backend via `GET /api/v1/translations?lang=`. No hardcoded copy in frontends.
- **Fallback**: English when a translation key is missing for the requested language
- **Frontend Integration**: Both apps wrap the UI in an `I18nProvider` that exposes a `t(key)` function and an RTL boolean

### Authentication

- **JWT Bearer Tokens**: Issued on login/register, sent in `Authorization` header
- **Password Hashing**: bcrypt
- **Password Reset**: Token-based flow (email sending is stubbed in development)
- **Protected Routes**: Frontend auth guards redirect unauthenticated users to login
- **Persistence**: Token stored in localStorage (web) and AsyncStorage (mobile)

### Dark Mode

- **Web**: Tailwind `class` strategy -- a `dark` class on `<html>` toggles all `dark:` variants
- **Mobile**: React Native `Appearance` API with user override stored in AsyncStorage
- **User Setting**: Persisted in `UserSettings.theme` on the backend

### Code-Based Identifiers

All entities use stable string codes as identifiers:
- `user_code`, `place_code`, `group_code`, `review_code`, `check_in_code`, `notification_code`, `attribute_code`
- Used in database primary/foreign keys, API paths, request/response bodies, and frontend types
- May include prefixes for readability (e.g., `usr_abc12`) but treated as opaque strings

### Religion Support

Three religions with distinct visual and data treatments:

| Religion      | Place Types                    | Specific Attributes                |
|---------------|--------------------------------|------------------------------------|
| Islam         | Mosques, shrines               | Prayer times, Jummah times         |
| Christianity  | Churches, cathedrals, shrines  | Service times, denomination         |
| Hinduism      | Temples, shrines               | Deities, festivals                  |

Users select their religion(s) of interest in `UserSettings.religions`, which filters the default place list.

### Feature Parity (Web and Mobile)

Both frontends maintain the same:
- Screen/route inventory and navigation structure
- API client methods (identical endpoint coverage)
- State management patterns (same four context providers)
- User flows (auth, browsing, check-in, reviews, groups, notifications)

Implementation differs only in platform primitives (React DOM + Tailwind vs React Native + Expo).

## 7. Data Flow

### User Registration and Authentication

```
Client (web/mobile)
  |-- POST /api/v1/auth/register  { email, password, display_name }
  |-- Server hashes password (bcrypt), generates user_code
  |-- Returns JWT token + user object
  |-- Client stores token, sets Authorization header for subsequent requests
```

### Place Discovery

```
Client
  |-- GET /api/v1/places?religion=islam&lat=21.4&lng=39.8&sort=distance
  |-- Server queries Place table, joins PlaceAttribute for attributes
  |-- Calculates Haversine distance if coordinates provided
  |-- Returns paginated list with images, attributes, open/closed status
```

### Data Scraping and Sync

```
Operator
  |-- POST /data-locations  (configure a Google Maps region or Google Sheet)
  |-- POST /runs            (start scraping)
  |-- Scraper queries Google Maps API in grid pattern within geo boundaries
  |-- Stores raw data in ScrapedPlace table
  |-- POST /runs/{code}/sync
  |     |-- For each ScrapedPlace:
  |     |     POST main-server/api/v1/places  (create or update)
  |     |-- Main server deduplicates by place_code
```

### Check-In Flow

```
Client
  |-- POST /api/v1/places/{code}/check-in  { note, photo_url }
  |-- Server creates CheckIn record
  |-- Server creates Notification for group members (if applicable)
  |-- Client refreshes check-in list, stats, and group leaderboard
```

### Group Invite Flow

```
Creator
  |-- POST /api/v1/groups  { name, path_place_codes, is_private }
  |-- Server generates invite_code
  |-- Creator shares invite_code or link

Invitee
  |-- POST /api/v1/groups/join-by-code  { invite_code }
  |-- Server creates GroupMember record
  |-- Server creates Notification for group members
```

### Translation Loading

```
Client (on app start)
  |-- GET /api/v1/translations?lang=en  (or ar, hi based on user setting)
  |-- I18nProvider stores translation map
  |-- I18nReadyGate / SplashScreen waits until loaded
  |-- All UI strings rendered via t("key") function
```
