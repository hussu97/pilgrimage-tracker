# Pilgrimage Tracker – Implementation Prompts

Use these prompts **one by one** to implement the system. Each prompt is a self-contained slice: backend + frontend for a set of screens or features. Reference [ARCHITECTURE.md](ARCHITECTURE.md) for stack and data model and [DESIGN_FILE.html](DESIGN_FILE.html) / [app-design-prompt-google-stitch.md](app-design-prompt-google-stitch.md) for UI and flows.

**Stack recap:** Monorepo with `apps/web` and `apps/mobile` (replicated frontend; no shared packages); React (Vite) + Tailwind in each; Capacitor for `apps/mobile`. Backend: Node/Express (or Fastify) + TypeScript, PostgreSQL, JWT auth. **APIs are versioned** (e.g. `/api/v1/...`). **All models and APIs use codes** (e.g. `place_code`, `user_code`), not numeric IDs; codes are autogenerated and may have a prefix/suffix for readability — do not use prefix/suffix in business logic. A Cursor rule enforces replicating UI and features in both web and mobile.

## Prompt 1: Project setup and shell

**Goal:** Create the monorepo, backend skeleton, and frontend shells for web and mobile so all later prompts can add features without changing structure. No shared `packages` folder; each app has its own types and API client.

**Tasks:**

1. **Monorepo and Cursor rule**
   - Create `apps/web` (Vite + React + TypeScript) and `apps/mobile` (same stack, Capacitor-ready). Create `server` (Express/Fastify + TypeScript). Do **not** create a `packages/` folder.
   - Add a **Cursor rules file** (e.g. `.cursor/rules/frontend-replication.mdc` or `frontend-replication.mdr`) with this rule: **"When adding or changing UI or features in `apps/web`, replicate the same UI and behavior in `apps/mobile`. When changing `apps/mobile`, replicate in `apps/web`. Keep screens, components, and feature behavior in sync; only app-specific config (e.g. Capacitor, env) may differ."**
   - In **both** `apps/web` and `apps/mobile`, add a local `src/types` (or similar) with placeholder types: `User` (with `user_code`), `Place` (with `place_code`), `Religion` enum.
   - In **both** apps, add a local API client that reads the API base URL (e.g. `VITE_API_URL`) and exposes a function like `getPlaces()` that returns `Promise<Place[]>` (mock data is fine). Client should call **versioned** API (e.g. `/api/v1/places`).
   - In **both** apps, add Tailwind with theme matching DESIGN_FILE.html: primary color, Lexend font, Material Icons and Material Symbols, border radius tokens. Add a simple layout with a header and a main area.

2. **Backend**
   - In `server`, set up Express (or Fastify), CORS, and a health route `GET /health` returning `{ status: "ok" }`.
   - Add a **versioned** route `GET /api/v1/places` that returns an empty array or 1–2 mock places (JSON) with a `place_code` on each (e.g. `plc_abc1`). No DB yet.
   - Document in README how to run `server`, `apps/web`, and `apps/mobile`, and that both apps use the same API base URL for `/api/v1`.

3. **Frontend shell**
   - In **both** `apps/web` and `apps/mobile`, set up React Router with routes: `/` (splash), `/login`, `/register`, `/select-path`, `/home`. Each route renders a placeholder page (e.g. a heading and a link to the next screen).
   - Add a responsive layout component in **both** apps: on viewport &lt; 768px use a bottom nav (Explore, Saved, Pilgrimage/Groups, Profile); on larger viewports use a top nav or sidebar. Use safe-area padding where DESIGN_FILE uses `safe-area-top` / `safe-area-bottom`.
   - Ensure both apps load without errors and navigation works. Routes that will later use entity codes should use `:placeCode`, `:groupCode`, etc. (e.g. `/places/:placeCode`).

**Done when:** Monorepo runs, API returns mock places with `place_code`, both web and mobile have routing and layout; Cursor rule file exists and states replication requirement.

## Prompt 2: Authentication (register, login, religion selection)

**Goal:** Implement registration, login, religion selection (onboarding), and a minimal “forgot password” flow so users can sign up and be identified.

**Tasks:**

1. **Backend**
   - Add PostgreSQL (or SQLite for local dev) and a `users` table: **user_code** (PK, autogenerated, e.g. prefix `usr_` + random), email, password_hash, display_name, religion (nullable enum: islam, hinduism, christianity), avatar_url, created_at, updated_at.
   - Implement `POST /api/v1/auth/register` (email, password, display_name); hash password (bcrypt), generate user_code, create user, return JWT (response includes user_code). Do not use code prefix/suffix in business logic.
   - Implement `POST /api/v1/auth/login` (email, password); validate, return tokens.
   - Implement `GET /api/v1/users/me` (protected): return current user from JWT (include user_code).
   - Implement `PATCH /api/v1/users/me` to set `religion` (for onboarding or later).
   - Add `POST /api/v1/auth/forgot-password` (email): create a reset token, store expiry, send email (or log link for dev). Add `POST /api/v1/auth/reset-password` (token, newPassword): validate token, update password.
   - In **each** app (apps/web, apps/mobile), add full User (with user_code) and auth types in local types; in local API client add register, login, getMe, updateMe, forgotPassword, resetPassword and attach JWT to requests.

2. **Frontend**
   - **Splash:** Match DESIGN_FILE “Splash & Welcome Screen”: logo, tagline “Discover, Visit, and Track Sacred Spaces”, “Get Started” (navigate to `/register`) and “Sign In” (navigate to `/login`).
   - **Register:** Match “Create Your Account”: full name, email, password, confirm password, validation messages, “Or continue with” Apple/Google (buttons only; no backend yet). On submit call local api client register, then redirect to `/select-path`.
   - **Login:** Email, password, “Forgot password?” link to `/forgot-password`. On submit call login, then redirect to `/home`.
   - **Forgot password:** Email input, “Send reset link”. Call `forgotPassword`; show success/error. Optional: simple reset-password page (token in query) that calls `resetPassword` and redirects to login.
   - **Religion selection (Select Your Path):** Match DESIGN_FILE “Select Your Path”: cards for Islam, Hinduism, Christianity; “Skip for now” and “View More Faiths” (optional). On select (or skip) call `updateMe({ religion })` then redirect to `/home`. If user already has religion, can redirect from `/select-path` to `/home`.
   - Persist auth: store tokens and set user (with `user_code`) in context; protect routes so unauthenticated users go to splash or login. Implement the same flows in **both** web and mobile (per Cursor rule).

**Done when:** User can register, select religion, log in, and be sent to home; `GET /api/v1/users/me` returns the user (with `user_code`); forgot/reset password works in dev. Same flows work in both web and mobile.

## Prompt 3: Home and discovery (list view, place cards)

**Goal:** Homepage shows religious places for the user’s religion, sorted by proximity; list view with place cards matching DESIGN_FILE “Explore Sacred Places”.

**Tasks:**

1. **Backend**
   - Add `places` table: **place_code** (PK, autogenerated, e.g. prefix `plc_` + random), name, religion, place_type, lat, lng, address, opening_hours (JSON), image_url, description, created_at. Add a few seed places (mix of religions and types).
   - Implement `GET /api/v1/places?religion=&lat=&lng=&limit=&offset=`: filter by religion (from query or from user’s profile if logged in), sort by distance when lat/lng provided; return list with place_code, name, religion, place_type, lat, lng, address, image_url, distance (computed). Optionally support `open_now` filter from opening_hours.
   - In **each** app, add Place (with place_code) and query types in local types; in local API client add getPlaces(params).

2. **Frontend**
   - **Home route:** If no user or no religion, redirect to splash or select-path. Otherwise show home.
   - **Homepage layout:** Match “Explore Sacred Places”: header with greeting (“Assalamu Alaikum, Ahmed” or “Welcome, [name]”) and religion context; List/Map toggle (list only in this prompt); search bar and filter chips (e.g. Nearby, Historical, “Jummah Prayer”, “Women’s Section”) – filter chips can be UI-only and hook to query params later.
   - **List view:** Fetch places with user’s religion and browser/device lat/lng (or default). Render place cards as in DESIGN_FILE: image, name, location, distance, rating (mock if needed), “Visited” badge if user has checked in (stub: assume no check-ins for now), “View Details” and “Check In” / “Bookmark” buttons. Tapping a card navigates to `/places/:placeCode` (placeholder in this prompt).
   - Use shared design tokens (Tailwind), Lexend, Material Icons. Responsive: cards stack on mobile; on desktop use a grid.
   - Bottom nav: Explore (active), Saved, Pilgrimage/Groups, Profile – links only.

**Done when:** Home shows places for the user’s religion sorted by distance; cards match design; clicking a card goes to place detail route (placeholder).

## Prompt 4: Place detail (religion-specific specs, opening times, reviews preview)

**Goal:** Place detail screen with hero, opening times, religion-specific content (deities for Hindu, prayer/facilities for Islam, denomination/service times for Christianity), and reviews preview.

**Tasks:**

1. **Backend**
   - Extend `places` with religion-specific JSON (e.g. `deities[]`, `festival_dates`, `dress_code` for Hindu; `prayer_times`, `capacity`, `facilities` for Islam; `denomination`, `service_times`, `notable_features` for Christianity). Seed a few places with this data.
   - Implement `GET /api/v1/places/:placeCode`: return full place including opening_hours and religion-specific fields. Return 404 when not found.
   - Add `reviews` table: **review_code** (PK), user_code (FK), place_code (FK), rating, title, body, created_at. Implement `GET /api/v1/places/:placeCode/reviews?limit=5`: return recent reviews with user display_name and rating. Optionally return aggregate rating/count on place.
   - In **each** app, add PlaceDetail, Review (with review_code, place_code) in local types; in local API client add getPlace(placeCode), getPlaceReviews(placeCode).

2. **Frontend**
   - **Place detail page** `/places/:placeCode`: Match DESIGN_FILE “Place Details - Hindu Temple” / “Place Details - Mosque” / “Place Details - Christian Church” as appropriate for `place.religion`.
   - Common: hero image and name; back/share/more actions; opening times (today + weekly if available); distance; “About” description; sticky bottom bar: “Get Directions”, “Check-in Here”, “Add to favorites” (actions can be stubs).
   - Religion-specific sections:
     - **Hinduism:** “Main Deities” (e.g. deity cards with image/name), “Temple Info” (architecture, next festival, dress code).
     - **Islam:** Prayer times, capacity, facilities (e.g. ablution).
     - **Christianity:** Denomination, service times, notable features (cards or list).
   - “Visitor Reviews”: show average rating and a few reviews (avatar, name, date, rating, text); “Write a review” link to `/places/:placeCode/review` (placeholder or next prompt).
   - Use one PlaceDetail component that branches on `place.religion` to render the right sections. Support dark mode if DESIGN_FILE specifies it for this screen.

**Done when:** Place detail loads for a given placeCode, shows opening times and religion-specific content, and a reviews preview; design matches DESIGN_FILE for the three religions. Same in both web and mobile (per Cursor rule).

## Prompt 5: Check-in and user profile

**Goal:** User can check in at a place; profile shows stats (e.g. places visited) and sections: visited places, favorites, groups, reviews.

**Tasks:**

1. **Backend**
   - Add `check_ins` table: **check_in_code** (PK), user_code (FK), place_code (FK), checked_in_at, note, photo_url (optional). Unique constraint or logic: one check-in per user per place per day (or allow multiple; decide product rule).
   - Implement `POST /api/v1/places/:placeCode/check-in` (body: optional note, optional photo): create check-in, return created record. Implement `GET /api/v1/users/me/check-ins`: return user’s check-ins with place summary.
   - Implement `GET /api/v1/users/me/stats`: return e.g. `{ placesVisited: number, checkInsThisYear: number }`.
   - In **each** app, add check-in and stats types and API client calls. Ensure `GET /api/v1/places` or place detail can include “user has checked in” for the current user (for “Visited” badge).

2. **Frontend**
   - **Check-in flow:** From place detail, “Check-in Here” opens a modal or navigates to a small screen: optional note/photo, confirm. On submit call `POST /api/v1/places/:placeCode/check-in`; show success (“You’ve checked in at [name]”) and close or go back. Update place card on home to show “Visited” when user has checked in.
   - **Profile page** `/profile`: Match DESIGN_FILE “User Profile & Stats”. Avatar, display name, email (or editable). Stats: “X religious sites visited”, “Y check-ins this year” (or similar). Sections: “Visited places” (list or grid of places with check-in date), “Favorites”, “My groups”, “Reviews I wrote” – each can be a link to a list or inline; implement visited places list in this prompt, others as links to placeholders.
   - **Edit profile** `/profile/edit`: Photo (upload stub or URL), display name, religion (dropdown), optional bio. PATCH to `/api/v1/users/me`. Save/cancel.
   - Profile and edit profile use same design tokens and layout (e.g. bottom nav).

**Done when:** User can check in from place detail; profile shows stats and visited places; edit profile updates name and religion; “Visited” appears on home cards where applicable.

## Prompt 6: Groups (list, create, detail, leaderboard, invite)

**Goal:** Users can create and join groups, see a leaderboard of places visited, and view recent group activity (check-ins).

**Tasks:**

1. **Backend**
   - Add `groups` table: **group_code** (PK, autogenerated), name, description, created_by_user_code (FK), invite_code (for joining), is_private, created_at. Add `group_members`: group_code (FK), user_code (FK), role (admin/member), joined_at. Add `group_invites`: group_code (FK), email or link, expires_at, used_at.
   - Implement `GET /api/v1/groups`, `POST /api/v1/groups`, `GET /api/v1/groups/:groupCode`, `PATCH /api/v1/groups/:groupCode`, `POST /api/v1/groups/:groupCode/join`, `POST /api/v1/groups/:groupCode/invite`, `GET /api/v1/groups/:groupCode/leaderboard`, `GET /api/v1/groups/:groupCode/activity`. Request/response bodies use group_code, user_code.
   - In **each** app, add Group, Member, LeaderboardEntry, ActivityItem (with group_code, user_code) in local types and API client.

2. **Frontend**
   - **Groups list** `/groups`: Match DESIGN_FILE “My Pilgrimage Groups”. “My Groups” header; featured card (e.g. “Most Active” with progress); list of groups with name, member count, progress (e.g. X/Y sites visited), “Last active”. FAB “Create group”. Tapping a group goes to `/groups/:groupCode`.
   - **Create group** `/groups/new`: Form: name, description, optional is_private. On submit call `POST /api/v1/groups`; redirect to group detail. Show “Invite” (copy link or share) using invite_code.
   - **Group detail** `/groups/:groupCode`: Match “Group Leaderboard & Activity”. Header with group name and member count; “Invite” button. Top 3 leaderboard (podium style) and full list (e.g. #4, #5, “View Full Leaderboard”). “Recently Visited” feed: activity items (e.g. “Amira K. checked in at Golden Temple”, “John D. reached a milestone – Notre Dame”). Use `GET /api/v1/groups/:id/leaderboard` and `GET /api/v1/groups/:id/activity`.
   - **Join flow:** From invite link (e.g. `/join?code=xxx`), call `POST /api/v1/groups/:id/join` and redirect to group detail.
   - Bottom nav: Groups tab active on groups list and group detail.

**Done when:** User can create a group, see list, open detail with leaderboard and activity feed, and join via invite link.

## Prompt 7: Favorites, settings, and notifications

**Goal:** Favorites (saved places), app settings, and a notifications list.

**Tasks:**

1. **Backend**
   - Add `favorites` table: user_code (FK), place_code (FK) — composite PK. Implement `POST /api/v1/places/:placeCode/favorite`, `DELETE /api/v1/places/:placeCode/favorite`, `GET /api/v1/users/me/favorites`: list places (with place_code).
   - Add `notifications` table: **notification_code** (PK, autogenerated), user_code (FK), type, payload (JSON), read_at, created_at. Implement `GET /api/v1/notifications` (paginated), `PATCH /api/v1/notifications/:notificationCode/read`. Create notifications when: group invite, someone in group checks in (optional), etc.
   - Settings: optional `GET/PATCH /api/v1/users/me/settings` (e.g. notifications_on, theme, units) or store in user row.

2. **Frontend**
   - **Favorites** `/favorites`: Match “Favorites / saved places” from design prompt. List of saved places (same card style as home); remove from favorites. On place detail, “Add to favorites” toggles saved state and calls add/remove API.
   - **Settings** `/settings`: Match “Settings screen”: notifications (check-in reminders, group updates), privacy (who can see check-ins – optional), language, theme (light/dark/system), units (km/miles), About and Terms links, Delete account (confirmation; call backend to delete or deactivate). Persist theme in localStorage and apply class or CSS vars.
   - **Notifications** `/notifications`: List of notifications (icon, title, body, time); mark as read on open or via button. Empty state when none.
   - Wire bottom nav “Saved” to `/favorites`. Add settings link from profile; notifications icon in header (with unread count badge if desired).

**Done when:** User can save/unsave places and see them on Favorites; settings screen works and theme persists; notifications list loads and can be marked read.

## Prompt 8: Reviews (write and list) and share flows

**Goal:** Users can write a review for a place; full reviews list on place detail; share place, check-in, or group invite.

**Tasks:**

1. **Backend**
   - Already have `reviews` (review_code, user_code, place_code). Implement `GET /api/v1/places/:placeCode/reviews`, `POST /api/v1/places/:placeCode/reviews` (rating, title, body, optional photos). `PATCH/DELETE /api/v1/reviews/:reviewCode` (only author). Return aggregate rating/count on place or in place detail response.
   - Optional: image upload for review photos (reuse file storage from profile avatar).

2. **Frontend**
   - **Write review** `/places/:id/review`: Match “Write a Review”. Form: star rating, optional title, body, optional photos. Submit calls `POST /api/v1/places/:placeCode/reviews`; success then redirect to place detail. Show validation errors.
   - **Place detail:** “Visitor Reviews” section: “View all” links to `/places/:id/reviews` (full list). On place detail, show aggregate rating and recent reviews; “Write a review” button for users who haven’t reviewed (or allow one per user and show “Edit” if they have).
   - **Share:** “Share” on place detail, check-in success, or group invite: use Web Share API when available (navigator.share) with title and URL (e.g. deep link or web URL); fallback copy link or open share sheet in Capacitor. URLs: place `/places/:placeCode`, group invite `/join?code=xxx`.

**Done when:** User can submit and edit/delete their review; place detail shows full reviews list and aggregate; share uses native share when possible.

## Prompt 9: Map view and search/filters

**Goal:** Map view of places on home (and optional dedicated map page); search and filters applied to both list and map.

**Tasks:**

1. **Backend**
   - Ensure `GET /api/v1/places` supports all query params: religion, lat, lng, radius (km), place_type, open_now, sort (proximity, rating), search (name or address). Add text search if needed (e.g. ILIKE or full-text).

2. **Frontend**
   - **Map view:** On home, when “Map” is selected, show a map (Mapbox GL or Leaflet) with pins for the same places returned by the list. Use user’s location as center if allowed; otherwise default center. Pins styled by place_type or religion. Tap pin: show small preview (name, distance); tap preview or “View details” to go to `/places/:placeCode`.
   - **Search:** Search bar on home: debounced input; update query params and refetch places (name/address search). Optional dedicated search page with filters.
   - **Filters:** Filter chips (Nearby, Historical, “Jummah Prayer”, “Women’s Section”, etc.) map to API params (e.g. place_type, tags). “Open now” uses opening_hours. Apply filters to both list and map; keep list/map toggle so both views use the same data.
   - Match DESIGN_FILE “Map Discovery View” for map styling and faith-specific pins if specified.

**Done when:** List and map show the same filtered places; search and filter chips update results; map pins open place preview then detail.

## Prompt 10: Polish (empty/error states, responsive desktop, optional PWA)

**Goal:** Consistent empty and error states; desktop layout polish; optional PWA and Capacitor build.

**Tasks:**

1. **Frontend**
   - **Empty states:** Per design prompt: no places nearby, no check-ins yet, no groups, no favorites. Each screen has a clear message and CTA (e.g. “Explore places”, “Create a group”).
   - **Error states:** No network, failed to load places, failed to check-in. Show message and “Retry” where appropriate. Use React Query or similar for retry and error UI.
   - **Responsive desktop:** On large viewports, use multi-column grids for lists, optional sidebar for nav, max-width content area. Ensure place detail and profile look good on desktop.
   - **Accessibility:** Focus order, aria-labels for icons, contrast. Ensure bottom nav and key actions are keyboard and screen-reader friendly.

2. **Optional**
   - **PWA:** Add manifest and service worker (e.g. Vite PWA plugin) for “Add to home screen” and offline shell.
   - **Capacitor:** Add Capacitor to the project; point it at the web build. Configure iOS and Android; add icons and splash. Test geolocation and (if used) camera for check-in photo. Build and run on device/simulator.

**Done when:** All main screens have appropriate empty and error UIs; desktop layout is polished; optional PWA/Capacitor works for mobile apps.

## Order of use

Execute prompts in order **1 → 10**. Each prompt assumes the previous ones are done (e.g. Prompt 3 assumes auth and shell from 1 and 2). You can break a prompt into smaller steps (e.g. backend first, then frontend) when implementing.

After implementation, the architecture is documented in [ARCHITECTURE.md](ARCHITECTURE.md) and this file holds the full list of implementation prompts for reference.
