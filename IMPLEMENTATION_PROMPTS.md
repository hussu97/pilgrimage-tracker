# Pilgrimage Tracker – Implementation Prompts

Use these prompts **one by one** to implement the system. Each prompt is a self-contained slice: backend + frontend for a set of screens or features. Reference [ARCHITECTURE.md](ARCHITECTURE.md) for stack and data model and [DESIGN_FILE.html](DESIGN_FILE.html) / [app-design-prompt-google-stitch.md](app-design-prompt-google-stitch.md) for UI and flows.

**Frontend and design:** For all frontend-related screens and UI, take **major inspiration from [DESIGN_FILE.html](DESIGN_FILE.html)**. Use it as the primary visual and layout reference: structure, components, spacing, typography (e.g. Lexend), colors, icons (Material Icons/Symbols), and screen flows. Where a prompt says “Match DESIGN_FILE …”, implement that screen to align closely with the corresponding section in DESIGN_FILE.html. Replicate the same look and behavior in both `apps/web` and `apps/mobile`.

**Stack recap:** Monorepo with `apps/web` and `apps/mobile` (replicated frontend; no shared packages); React (Vite) + Tailwind in each; Capacitor for `apps/mobile`. **Backend: Python 3.11+ and FastAPI** (Uvicorn), Pydantic schemas, JWT auth; PostgreSQL or SQLite for local dev. **APIs are versioned** (e.g. `/api/v1/...`). **All models and APIs use codes** (e.g. `place_code`, `user_code`), not numeric IDs; codes are autogenerated and may have a prefix/suffix for readability — do not use prefix/suffix in business logic. A Cursor rule enforces replicating UI and features in both web and mobile.

## Migration: Node.js backend → Python/FastAPI (Prompts 1–3 and 3.5)

If you **already implemented Prompts 1, 2, 3 (and any custom Prompt 3.5)** with the previous Node.js/Express backend, use this section to migrate the backend only. The **frontend does not need to change** as long as the API contract (URLs, methods, request/response JSON) stays the same.

### Backend changes required

- **Replace** the existing `server/` (Node.js, Express, TypeScript) with a new **Python + FastAPI** application in `server/`.
- **Remove:** Node/Express/TS runtime, `package.json`, `tsconfig.json`, and all `.ts` files under `server/`.
- **Add:** Python project under `server/`:
  - `requirements.txt`: `fastapi`, `uvicorn[standard]`, `pydantic`, `python-jose[cryptography]`, `passlib[bcrypt]`, `python-multipart`; optional: `sqlalchemy`, `databases`, `asyncpg` or `aiosqlite`.
  - `app/main.py`: FastAPI app, CORS middleware, include API router under prefix `/api/v1`.
  - `app/core/config.py`: settings (e.g. `SECRET_KEY`, `DATABASE_URL`).
  - `app/core/security.py`: password hashing (passlib), JWT create/verify (python-jose).
  - `app/db/store.py` or `app/db/session.py`: in-memory store (dicts/lists) or SQLAlchemy; same data shape as current (users with `user_code`, places with `place_code`).
  - `app/models/`: Pydantic models for request/response (e.g. `UserResponse`, `PlaceList`, `Token`).
  - `app/api/v1/`: routers for `auth`, `users`, `places` — same paths and response shapes as the Node API (e.g. `POST /api/v1/auth/register` returns `{ user, token }`; `GET /api/v1/places` returns list of objects with `place_code`, etc.).
- **Reimplement** in Python:
  - Auth: register (hash password, generate `user_code`, return JWT), login, get current user, update religion, forgot/reset password (store reset tokens, same expiry semantics).
  - Places: list with optional `religion`, `lat`, `lng`, `limit`, `offset`; compute distance when lat/lng given; return list with `place_code`, name, religion, place_type, lat, lng, address, image_urls, distance.
- **Run:** `uvicorn app.main:app --reload --port 3000` (or keep port 8000 and point frontend proxy/env to it).

### Frontend changes required

- **None** if the new FastAPI backend exposes the same routes and JSON shapes. Only ensure the web/mobile apps call the correct base URL (e.g. `VITE_API_URL` or Vite proxy target). If the Python server runs on port 8000 instead of 3000, update the proxy in `apps/web/vite.config.ts` and `apps/mobile/vite.config.ts` (e.g. `target: 'http://localhost:8000'`) or set `VITE_API_URL=http://localhost:8000`.

### Prompt 3.5 (if you have one)

If you added a custom **Prompt 3.5** between 3 and 4, migrate its backend tasks to FastAPI as well: same API paths and response bodies, implemented in Python with Pydantic and FastAPI routers.

## Prompt 1: Project setup and shell

**Goal:** Create the monorepo, backend skeleton, and frontend shells for web and mobile so all later prompts can add features without changing structure. No shared `packages` folder; each app has its own types and API client.

**Tasks:**

1. **Monorepo and Cursor rule**
   - Create `apps/web` (Vite + React + TypeScript) and `apps/mobile` (same stack, Capacitor-ready). Create `server` (Python + FastAPI). Do **not** create a `packages/` folder.
   - Add a **Cursor rules file** (e.g. `.cursor/rules/frontend-replication.mdc` or `frontend-replication.mdr`) with this rule: **"When adding or changing UI or features in `apps/web`, replicate the same UI and behavior in `apps/mobile`. When changing `apps/mobile`, replicate in `apps/web`. Keep screens, components, and feature behavior in sync; only app-specific config (e.g. Capacitor, env) may differ."**
   - In **both** `apps/web` and `apps/mobile`, add a local `src/types` (or similar) with placeholder types: `User` (with `user_code`), `Place` (with `place_code`), `Religion` enum.
   - In **both** apps, add a local API client that reads the API base URL (e.g. `VITE_API_URL`) and exposes a function like `getPlaces()` that returns `Promise<Place[]>` (mock data is fine). Client should call **versioned** API (e.g. `/api/v1/places`).
   - In **both** apps, add Tailwind with theme matching DESIGN_FILE.html: primary color, Lexend font, Material Icons and Material Symbols, border radius tokens. Add a simple layout with a header and a main area.

2. **Backend (Python + FastAPI)**
   - In `server/`, set up a FastAPI app (e.g. `app/main.py`), CORS middleware, and a health route `GET /health` returning `{ "status": "ok" }`. Run with Uvicorn.
   - Add a **versioned** route `GET /api/v1/places` that returns an empty array or 1–2 mock places (JSON) with a `place_code` on each (e.g. `plc_abc1`). Use a Pydantic response model; no DB yet.
   - Document in README how to run `server` (e.g. `uvicorn app.main:app --reload`), `apps/web`, and `apps/mobile`, and that both apps use the same API base URL for `/api/v1`.

3. **Frontend shell**
   - In **both** `apps/web` and `apps/mobile`, set up React Router with routes: `/` (splash), `/login`, `/register`, `/select-path`, `/home`. Each route renders a placeholder page (e.g. a heading and a link to the next screen).
   - Add a responsive layout component in **both** apps: on viewport &lt; 768px use a bottom nav (Explore, Saved, Pilgrimage/Groups, Profile); on larger viewports use a top nav or sidebar. Use safe-area padding where DESIGN_FILE uses `safe-area-top` / `safe-area-bottom`.
   - Ensure both apps load without errors and navigation works. Routes that will later use entity codes should use `:placeCode`, `:groupCode`, etc. (e.g. `/places/:placeCode`).

**Done when:** Monorepo runs, API returns mock places with `place_code`, both web and mobile have routing and layout; Cursor rule file exists and states replication requirement.

## Prompt 2: Authentication (register, login, religion selection)

**Goal:** Implement registration, login, religion selection (onboarding), and a minimal “forgot password” flow so users can sign up and be identified.

**Tasks:**

1. **Backend (Python + FastAPI)**
   - In `server/`, add a users store (in-memory dict or PostgreSQL/SQLite): **user_code** (PK, autogenerated, e.g. prefix `usr_` + random), email, password_hash, display_name, religion (nullable enum: islam, hinduism, christianity), avatar_url, created_at, updated_at. Use Pydantic models for request/response.
   - Implement `POST /api/v1/auth/register` (email, password, display_name): hash password (passlib + bcrypt), generate user_code, create user, return JWT (response includes user_code). Do not use code prefix/suffix in business logic.
   - Implement `POST /api/v1/auth/login` (email, password); validate, return tokens (e.g. python-jose or PyJWT).
   - Implement `GET /api/v1/users/me` (protected): return current user from JWT (include user_code); use a dependency that validates JWT and returns the user.
   - Implement `PATCH /api/v1/users/me/religion` (body: religion) for onboarding or later.
   - Add `POST /api/v1/auth/forgot-password` (email): create a reset token, store expiry, send email (or log link for dev). Add `POST /api/v1/auth/reset-password` (token, newPassword): validate token, update password.
   - In **each** app (apps/web, apps/mobile), add full User (with user_code) and auth types in local types; in local API client add register, login, getMe, updateMe, forgotPassword, resetPassword and attach JWT to requests.

2. **Frontend**
   - **Splash:** Match DESIGN_FILE “Splash & Welcome Screen”: logo, tagline “Discover, Visit, and Track Sacred Spaces”, “Get Started” (navigate to `/register`) and “Sign In” (navigate to `/login`).
   - **Register:** Match “Create Your Account”: full name, email, password, confirm password, validation messages, “Or continue with” Apple/Google (buttons only; no backend yet). On submit call local api client register, then redirect to `/select-path`.
   - **Login:** Email, password, “Forgot password?” link to `/forgot-password`. On submit call login, then redirect to `/home`.
   - **Forgot password:** Email input, “Send reset link”. Call `forgotPassword`; show success/error. Optional: simple reset-password page (token in query) that calls `resetPassword` and redirects to login.
   - **Religion selection (Select Your Path):** Match DESIGN_FILE “Select Your Path”: cards for Islam, Hinduism, Christianity; “Skip for now” and “View More Faiths” (optional). On select (or skip) call `PATCH /api/v1/users/me/religion` with selected religion, then redirect to `/home`. If user already has religion, can redirect from `/select-path` to `/home`.
   - Persist auth: store tokens and set user (with `user_code`) in context; protect routes so unauthenticated users go to splash or login. Implement the same flows in **both** web and mobile (per Cursor rule).

**Done when:** User can register, select religion, log in, and be sent to home; `GET /api/v1/users/me` returns the user (with `user_code`); forgot/reset password works in dev. Same flows work in both web and mobile.

Add a few seed places (mix of religions and types). Use Pydantic models for list/detail responses.

Optionally support `open_now` filter from opening_hours.

2. **Frontend**

   - **Homepage layout:** Match “Explore Sacred Places”: header with greeting (“Assalamu Alaikum, Ahmed” or “Welcome, [name]”) and religion context; List/Map toggle (list only in this prompt); search bar and filter chips (e.g. Nearby, Historical, “Jummah Prayer”, “Women’s Section”) – filter chips can be UI-only and hook to query params later.

Tapping a card navigates to `/places/:placeCode` (placeholder in this prompt).

**Done when:** Home shows places for the user’s religion sorted by distance; cards match design; clicking a card goes to place detail route (placeholder).

## Prompt 4: Place detail (religion-specific specs, opening times, reviews preview)

**Goal:** Place detail screen with hero, opening times, religion-specific content (deities for Hindu, prayer/facilities for Islam, denomination/service times for Christianity), and reviews preview.

**Tasks:**

1. **Backend (Python + FastAPI)**
   - In `server/`, extend places store/model with religion-specific JSON (e.g. `deities[]`, `festival_dates`, `dress_code` for Hindu; `prayer_times`, `capacity`, `facilities` for Islam; `denomination`, `service_times`, `notable_features` for Christianity). Seed a few places with this data. Use Pydantic models for place detail response.
   - Add a FastAPI router/endpoint `GET /api/v1/places/{place_code}`: return full place including opening_hours and religion-specific fields. Return 404 when not found (use HTTPException).
   - Add reviews store/table: **review_code** (PK), user_code (FK), place_code (FK), rating, title, body, created_at. Implement `GET /api/v1/places/{place_code}/reviews` with query param `limit` (e.g. default 5): return recent reviews with user display_name and rating. Optionally return aggregate rating/count on place or in place detail response.
   - In **each** app, add PlaceDetail, Review (with review_code, place_code) in local types; in local API client add getPlace(placeCode), getPlaceReviews(placeCode).

2. **Frontend**
   - **Place detail page** `/places/:placeCode`: Match DESIGN_FILE “Place Details - Hindu Temple” / “Place Details - Mosque” / “Place Details - Christian Church” as appropriate for `place.religion`.
   - Common: hero image and name; back/share/more actions; opening times (today + weekly if available); distance; “About” description; sticky bottom bar: “Get Directions”, “Check-in Here”, “Add to favorites” (actions can be stubs).
   - Religion-specific sections:
     - **Hinduism:** “Main Deities” (e.g. deity cards with image/name), “Temple Info” (architecture, next festival, dress code).
     - **Islam:** Prayer times, capacity, facilities (e.g. ablution).
     - **Christianity:** Denomination, service times, notable features (cards or list).
   - “Visitor Reviews”: show average rating and a few reviews (avatar, name, date, rating, text); “Write a review” link to `/places/:placeCode/review` (placeholder or next prompt).
   - Use one PlaceDetail component that branches on `place.religion` to render the right sections. Support dark mode if DESIGN_FILE specifies it for this screen.

**Done when:** Place detail loads for a given placeCode, shows opening times and religion-specific content, and a reviews preview; design matches DESIGN_FILE for the three religions. Same in both web and mobile (per Cursor rule).

## Prompt 5: Check-in and user profile

**Goal:** User can check in at a place; profile shows stats (e.g. places visited) and sections: visited places, favorites, groups, reviews.

**Tasks:**

1. **Backend (Python + FastAPI)**

   - Implement `POST /api/v1/places/{place_code}/check-in` (body: optional note, optional photo): create check-in, return created record. Implement `GET /api/v1/users/me/check-ins`: return the current user’s check-ins with place summary (use JWT dependency for current user).
   - Implement `GET /api/v1/users/me/stats`: return e.g. `{ "placesVisited": number, "checkInsThisYear": number }`.
   - In **each** app, add check-in and stats types and API client calls. Ensure `GET /api/v1/places` or place detail can include “user has checked in” for the current user (for “Visited” badge).

2. **Frontend**
   - **Check-in flow:** From place detail, “Check-in Here” opens a modal or navigates to a small screen: optional note/photo, confirm. On submit call `POST /api/v1/places/:placeCode/check-in`; show success (“You’ve checked in at [name]”) and close or go back. Update place card on home to show “Visited” when user has checked in.
   - **Profile page** `/profile`: Match DESIGN_FILE “User Profile & Stats”. Avatar, display name, email (or editable). Stats: “X religious sites visited”, “Y check-ins this year” (or similar). Sections: “Visited places” (list or grid of places with check-in date), “Favorites”, “My groups”, “Reviews I wrote” – each can be a link to a list or inline; implement visited places list in this prompt, others as links to placeholders.
   - **Edit profile** `/profile/edit`: Photo (upload stub or URL), display name, religion (dropdown), optional bio. Use `PATCH /api/v1/users/me` for profile fields and `PATCH /api/v1/users/me/religion` when religion changes. Save/cancel.
   - Profile and edit profile use same design tokens and layout (e.g. bottom nav).

**Done when:** User can check in from place detail; profile shows stats and visited places; edit profile updates name and religion; “Visited” appears on home cards where applicable.

## Prompt 10: Polish (empty/error states, responsive desktop, optional PWA)

**Goal:** Consistent empty and error states; desktop layout polish; optional PWA and Capacitor build.

   - **Empty states:** Per design prompt: no places nearby, no check-ins yet, no groups, no favorites. Each screen has a clear message and CTA (e.g. “Explore places”, “Create a group”).

**Done when:** All main screens have appropriate empty and error UIs; desktop layout is polished; optional PWA/Capacitor works for mobile apps.

## Order of use

Execute prompts in order **1 → 10**. Each prompt assumes the previous ones are done (e.g. Prompt 3 assumes auth and shell from 1 and 2). You can break a prompt into smaller steps (e.g. backend first, then frontend) when implementing.

After implementation, the architecture is documented in [ARCHITECTURE.md](ARCHITECTURE.md) and this file holds the full list of implementation prompts for reference.
